# ARO 探索档(explore mode)— 设计方案(待 review)

> 把 ARO 从「证明一个可合的赢」升级为「**无限尝试 · 把所有路线画成树 · 人只在叶子上拍板**」。
> 模型在保留的测量严谨下广撒网探索整个项目,产出一棵分级的尝试树;人在叶子上决定「合 / 不合 / 加一条例子」,例子反过来给后续探索剪枝。**不追求穷尽,但保证大致方向都能走到。**

---

## 0. 这是什么、不是什么

- **是**:一个无限探索器 + 一张「试过的所有路线」的可审树 + 人在叶子拍板 + 例子库随反馈成长。
- **不是**:① 不产「可直接合」的版本(合不合、该不该这么做,**到叶子由人定**);② **不追求穷尽**(无限生成撞上有限的串行判分,做不到);③ 但**大致方向必须可达**(不能一直只挖最肥那块、别的方向永远轮不到)。

---

## 1. 保留的(moat 核心,一点不松)

1. **A/A + A/B 显著性**:A/A 噪声地板 + 配对 A/B(顺序交替)+ bootstrap CI 排 0 + 地板检查 + auto-tighten。**bench 永远串行**。
2. **正确性底线**:build + cargo test 必须过。
3. **真测出提升**:在 bench(现成或新写)上必须有**显著**的速度提升,A/A+A/B 证。
4. **Gate 0 反作弊红线**:写手永不碰尺子 —— 绝不编辑 `Cargo.toml`/lock、`benches/`、`tests/`。
5. **★ 独立语义评审(critic gate)**:**所有新产出 —— bench、代码、思路 —— 都要过一个独立的评审 subagent,过了才算。** 确定性 judge 证「数字」(快+正确),但判不了「这 bench 有没有被做手脚 / 这代码是不是 reward-hack / 这思路通不通」。这道由独立 LLM 评审(对抗式、默认怀疑)补上。**两道都过才进树。**(见 §3.0)
6. **★ 全程可回溯(audit trail)**:两道 judge 都**不只记「过/否」,还记「为什么」** —— 每个关节节点(思路/计划、bench、每个候选、每道 gate、每个评审、最终判定)的**判定内容 + 理由 + 数字**都 verbatim 进 events.jsonl,并在树的叶子上拼成一条完整的「为什么过 / 为什么没过」链路。**任何结论都能一路回溯到证据。**(见 §3.7)

---

## 2. 放松的(换来探索面 + 减人门)

| 维度 | 严格档 | 探索档 | 代价(有意识接受) |
|---|---|---|---|
| 正确性 oracle | 字节相同 differential | **单测过即底线**;改到**无测试覆盖**的代码 → **自动生成 differential 兜底**(EVM 不裸奔) | accept ≠ 字节相同证明 |
| 范围 | 定位到的单文件 | **整个 workspace 的 src**(Gate 0 红线仍在) | 跨 crate 大改更多 |
| 维护性把关 | 人门每个改动 | **例子驱动**(软引导 + 硬剪枝);人只在叶子拍板 / 加例子 | 例子库要养 |
| 负载 | 人给 spec | 现成 bench,或 **agent 自写**(maker-checker) | 微基准代表性存疑 |
| 产出语义 | accepted = 可证明可合 | accepted = **线索,待人在叶子定夺** | 人做最终 triage |

## 3. 机制

### 3.0 ★ 独立语义评审(critic gate)—— 两道 judge
maker-checker 从「数字」延到「语义」:除了确定性 judge,每个新产出还要过一个**独立评审 subagent**(喂它评审 prompt),**过了才算**。三类产出各一套 rubric:
- **bench**:真的驱动到目标函数了吗?是不是 trivial / 可被轻易"刷"赢?隔离得对不对?(防自写尺子作弊。)
- **代码**:是真优化还是 **reward-hack / 钻 bench 空子**?有没有命中**已知坏 pattern**(`optimization-examples.md`,如溶解分层)?跨边界/弱 oracle 风险标了吗?
- **思路**(read-phase 计划 + 候选 hypothesis):推理站得住吗?依赖的不变量是真的吗?

**怎么放、怎么可信(关键)**:
- **放在串行 judge 之前**:评审是并行的 LLM 调用(跟生成一样并行),**只有评审通过的候选才进那条稀缺的串行 bench** —— 既守语义,又当智能预筛、省吞吐。它是 §3.4 例子硬剪枝的更强版本。
- **独立 + 对抗**:评审 agent ≠ 生成 agent(maker-checker);prompt 让它**默认怀疑、主动找茬**(「请尝试证明这是错的 / 被刷的 / 不该做的」),拿不准就**否**。
- **多票**(留口子,暂不做):现在**单评审**就够;接口设计成「N 个评审独立投票、多数过」可平滑接入(评审就是个 `critique(artifact) -> {verdict, reasons}` 的函数,N 票只是多调几次取多数)。**先单后多。**
- **不能放水**:评审是**额外**一道(AND,不是 OR)—— 候选必须**同时**过确定性 judge 和评审。评审能否、不能改判「数字」。
- **可审**:每个评审的「过/否 + 理由」**记进 events.jsonl、显示在树的叶子上** —— 人能看到「为什么被否」,也能据此**加例子**(喂回 §3.4)。
- **谁评审评审?** 评审自己是 LLM、会错。靠:① 独立 + 对抗默认否;② 多票;③ 理由 verbatim 上树,人最终在叶子复核。**评审是第二道 judge,不是真理。**

### 3.1 生成无限,判分串行(吞吐是真天花板)
生成可无限 fan-out;但每个候选都得过那条**串行 bench**。墙钟 = 进串行队列的候选数 × 单次 bench 时间。因此必做:
- **廉价预筛**:能 build? 与 baseline 真有差异? 一次 smoke bench 估粗 Δ。三关任一不过 → 不进串行队列。
- **去重**:等价改(同 AST/同文本)只判一次。
- **判分队列优先级**:按 smoke-Δ 从大到小判 —— 稀缺的判分时间先给最像赢的。
> 「无限」= 生成无限、判分按优先级尽力,**不是判分无限**。

### 3.2 范围 = 整个 workspace
`editable` 放宽到所有 workspace member 的 `src/`;Gate 0 红线(Cargo/lock、benches、tests)不变。改坏由 build+test+(differential)兜。

### 3.3 自写 bench + maker-checker + 无覆盖兜底
- 改的代码没有可测的 bench → **autonomous 自写一个 per-function 微基准**。
- **maker-checker**:写 bench 的 agent ≠ 写优化的 agent;bench **先冻结**再优化(防自写尺子作弊)。
- 改到**无测试覆盖**的函数 → **自动生成 differential 探针**(随机输入比指纹),否则「单测过」是真空的。

### 3.4 例子:软引导 + 硬剪枝
- **软**(生成侧):`optimization-examples.md` 的坏实践例子(如 PR#313 双 `inspect_storage` 溶解分层)注入 prompt。
- **硬**(判分前):候选与「已知坏 pattern」相似 → **自动降到队尾 / 跳过**,不浪费串行判分。
- **回路**:人在叶子说「这个不该做」→ 落一条新 example → 后续自动避开。**这是「减负担」真正生效的地方。**

### 3.5 ★ 方向可达:广度优先 + explore/exploit 预算(本方案的关键新机制)
纯按 smoke-Δ 贪心,会一直只挖最肥的那块(如存储路径),precompile / 资源限额 / dual-gas 等**永远轮不到** —— 这违背「大致方向要可达」。所以:
- **先广后深**:第一遍**每个 crate / 每类方向各打一击**(广度),找出哪里有戏;再在有戏处加深。
- **explore/exploit 配额(bandit / MCTS 式)**:判分队列**不纯贪心** —— 留一份预算给**最少被探索的方向**(新 crate、新手法、新档),即使它们 smoke-Δ 未知。这样**每个大方向都拿得到尝试名额**,无需穷尽。
- **覆盖轴(已定核心 4 条)**:配额按这几条轴铺,保证不偏食:
  1. **crate** —— 别只挖一个 crate(mega-evm / ipa-multipoint / banderwagon / salt-glue …,走 cargo metadata)。
  2. **手法(最重要)** —— 别只换 lens 档,要换**做法**:消冗余 / 缓存·记忆化 / 数据布局 / 消分配 / 批处理·融合 / 算法替换 / 分支重排。这条保证试了**不同种类**的改,而非同一种的不同档。
  3. **热度档** —— top>5% / 中 1-5% / 尾<1%,别只盯 #1 热点。
  4. **风险档** —— 字节相同 / 放宽·结构 / 跨 crate,安全赢与大胆探索都要有。
  - 可选更细轴:**来源**(自家 crate / 可改的 fork / 胶水)、**模块**(crate 内更细)。
> 这是「不穷尽但方向可达」的落地:**用 explore/exploit 配额按上述轴铺,保证覆盖,而不是靠走完整棵树。**

### 3.6 输出:分级可审树(就是 decision-tree.html)
无限探索 → 上千叶子,人审不过来。所以树**分级**:
- **顶部:值得你看的 N 个** —— 大赢优先,标 **risk**(跨 crate / 溶解分层 / 命中坏例子 / 弱 oracle)。
- **长尾:相似尝试聚类折叠**。
- 叶子点开:**完整的「为什么」链路**(见 §3.7)+ diff(紧凑彩色)+ Δ/CI + 维护性 flag,让人**几秒**做出「合 / 不合 / 加例子」的决定。

### 3.7 ★ 全程可回溯:每个关节节点都记「判定 + 理由」
两道 judge 都把**判定内容**留痕,叶子上拼成完整证据链。一个候选的可回溯路径:
1. **思路 / read-phase 计划** + 评审对思路的「过/否 + 理由」。
2. **bench**(若新写)+ 评审对 bench 的「过/否 + 理由」+ bench 自身 profile(证它真驱动到目标函数)。
3. **代码候选** + 评审对代码的「过/否 + 理由」(命中哪条坏例子、reward-hack 嫌疑…)。
4. **确定性各 gate**:guard / build / test / differential / 显著性,每道的 **status + detail + 数字**(Δ/CI/floor/scale、auto-tighten 记录)。
5. **最终判定** + 一句话归因。
- 全部 **verbatim 进 events.jsonl**(机器真相),树**只读不二次判分**(不变量 §6.4)。
- 评审的理由要**结构化记**(verdict / rubric 命中项 / 关键论据),不是一坨自由文本 —— 方便树上展示 + 以后「人加例子」时直接引用。
- 价值:① 人在叶子**几秒看懂为什么**;② 被否的能回溯、能据此**加反例**;③ 出问题(评审误杀/放水)能**事后审计到具体那条理由**。

---

## 4. 人的闭环(只在叶子)
对每个浮上来的候选,人三选一:**① 采纳(进持久 baseline)② 否决 ③ 加一条 example**(否决+说明为什么)。①②③ 都不打断探索;③ 喂回 3.4 给未来剪枝。

---

## 5. 停机(不穷尽)
- **预算**:token / 墙钟(token 无限时,墙钟由串行判分定)。
- **递减**:某方向连续 K 轮无新赢 → 降权该方向(但 3.5 的配额仍保证它**被探过**,只是不再深挖)。
- **方向可达性是配额保证的,不是穷尽保证的。**

---

## 6. 不变量(任何改动不能破)
1. bench 串行。2. 写手不碰尺子(**含 bench**)。3. 正确性先于显著性。4. **判定 + 理由 + 数字**全 verbatim 进 events.jsonl(两道 judge 都留痕、可回溯),树只读不二次判分。5. 通用性(owner/范围走 cargo metadata,不堆特例)。6. 两道 judge(确定性 + 语义评审)都过才进树;评审独立于写手、能否不能放水(AND 不是 OR)。

## 8. 落地阶段
- **横切(贯穿 A/B/C)**:**独立语义评审 critic gate** —— 一上来就接进判分前(代码+思路),bench 一写就评审。它是「两道 judge」的第二道,越早越好。
- **阶段 A**:critic gate(代码+思路)+ whole-project 范围 + explore/exploit 配额(方向可达)+ 树分级。← 让「广撒网、方向可达、且每个候选过语义评审」先跑起来。
- **阶段 B**:maker-checker 自写 bench + bench 评审 + 无覆盖自动 differential。← 让「自己补 bench」可信。
- **阶段 C**:例子硬剪枝 + 人在叶子「加例子」回路。← 让「减负担」真生效。
